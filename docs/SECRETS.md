# Secrets at rest

How `sentinel-harness` keeps credentials out of source, out of environment variables, and
out of the agent's reach. Short and concrete; every claim cites the code.

## The rule: the agent never sees a raw credential

Third-party tokens (a SIEM API key, a threat-intel token, an OAuth secret) are **never**
placed in plaintext environment variables, harness config, or source. Two mechanisms:

### 1. AgentCore Identity token vault (preferred for tools)

Tool configs reference a secret by **vault ARN interpolation** — `${arn:...}` — which
AgentCore Identity resolves **server-side** at invoke time. The raw secret is injected into
the outbound request by the platform; it never transits your config or the model context.

- `tool_remote_mcp(name, url, headers=...)` — header values may use `${arn:...}`
  interpolation ([`sentinel_harness/core.py`](../sentinel_harness/core.py)).
- The YAML loader expands ordinary `${ENV_VAR}` references but **deliberately leaves
  `${arn:...}` untouched** so vault interpolation is resolved by Identity, not the loader
  ([`sentinel_harness/loader.py`](../sentinel_harness/loader.py) — the `_ENV_REF` regex
  matches all-caps env names only, never the lowercase-`arn:` form).
- Gateway targets accept `credentialProviderConfigurations` so outbound auth is bound to a
  credential provider, not a literal token
  ([`sentinel_harness/gateway.py`](../sentinel_harness/gateway.py) `create_gateway_target`).

The execution role's grants for this path (`GetResourceApiKey`,
`GetResourceOauth2Token`, `GetWorkloadAccessToken*`) are in
[`docs/SETUP.md`](SETUP.md) — the agent's role can *fetch a resolved token for a call*, but
the token is not materialized into config or logs.

### 2. AWS Secrets Manager / SSM Parameter Store `SecureString`

For secrets your own code (a Lambda tool handler, a deploy script) needs directly, store
them in **Secrets Manager** or an **SSM `SecureString`** parameter and fetch at runtime with
the role's scoped permission. **Never** a plaintext env var, and never committed.

- Do: `secretsmanager:GetSecretValue` / `ssm:GetParameter` with `WithDecryption`, scoped to
  the specific secret/parameter ARN.
- Don't: `export SIEM_TOKEN=...` in a shell profile, a Dockerfile `ENV`, or a `.env` that
  can be committed.

## No secrets in the IaC either

The Cognito **machine (M2M) client secret** is generated by Cognito and lives
**server-side** — the CDK sets `generateSecret: true` and never emits the secret value as a
plaintext output; only non-secret ids (user pool id, client ids, discovery URL) are
`CfnOutput`s ([`iac-cdk/lib/identity-stack.ts`](../iac-cdk/lib/identity-stack.ts)). This is
part of what the live-verify checks as "zero plaintext secrets" (see the README's live-verify
row).

The Guardrail regex **patterns** for secret shapes are assembled from character-class
fragments so that **no literal credential** (no real `AKIA…`, no real `sk-…`/`ghp_…`) ever
sits in source and trips the repo secret-scanner
([`iac-cdk/lib/guardrail-stack.ts`](../iac-cdk/lib/guardrail-stack.ts)). A matcher is not a
secret. This mirrors the CI `secret-and-name` scan that fails the build on any hardcoded AWS
account id or access-key shape (`.github/workflows/ci.yml`).

## Rotation

- **Vault-backed tokens:** rotate the underlying secret in Secrets Manager (native
  rotation) or in the AgentCore Identity credential provider. Because tool configs reference
  the secret by ARN, **nothing in this repo changes on rotation** — no redeploy of harness
  config, no code edit. That indirection is the point.
- **Cognito M2M secret:** rotate by issuing a new app-client secret and updating the
  consuming credential provider; the pool/issuer/discovery URL are stable, so the Gateway
  authorizer config is unaffected.
- **Execution-role / caller credentials:** these are IAM role sessions (short-lived STS
  credentials), not long-lived keys — there is no static key to rotate. Do not create IAM
  users with long-lived access keys for this; use role assumption.

## Why the execution-role model omits `InvokeAgentRuntimeCommand`

This is a secrets *and* privilege decision, so it belongs here too.

`bedrock-agentcore:InvokeAgentRuntimeCommand` runs an arbitrary shell command on the
microVM **as root, bypassing the LLM and `allowedTools` entirely**. A process with shell +
root can read any secret the microVM can reach — environment, mounted files, instance
metadata, a resolved vault token in memory — regardless of every other control on this page.
`allowedTools` cannot restrict it (it only scopes what the *model* may pick during
`InvokeHarness`); the **only** control is to not grant the IAM action.

So the starter execution-role policy in [`docs/SETUP.md`](SETUP.md) **deliberately excludes**
it (see the `__comment__` in the policy JSON and the ⚠️ callout), and
[`docs/THREAT-MODEL.md`](THREAT-MODEL.md) §6 treats it as the single most important
least-privilege decision for a SecOps repo. Grant it only if a scenario genuinely needs
deterministic shell prep, and understand that doing so re-opens the full blast radius. Even
then, wrap it with the `sandbox_hooks` PreToolUse validators
([`sentinel_harness/sandbox_hooks.py`](../sentinel_harness/sandbox_hooks.py)).

## Checklist

- [ ] No third-party token in plaintext env, config, or source — use `${arn:...}` vault
  interpolation or Secrets Manager / SSM `SecureString`.
- [ ] Secret fetches scoped to the specific secret/parameter ARN, not `*`.
- [ ] `InvokeAgentRuntimeCommand` not granted (unless justified + sandbox-wrapped).
- [ ] IAM = role assumption / short-lived STS, no long-lived IAM user keys.
- [ ] CI `secret-and-name` scan green (no hardcoded account ids / access keys).

See [`SECURITY.md`](../SECURITY.md) to report a leak and
[`docs/THREAT-MODEL.md`](THREAT-MODEL.md) for the full control map.
